## Technische Analyse je Issue

### Issue #180: Schwellwert-Tabelle

**UX-Konzept:** Anders als der bestehende `AlertRulesEditor` (dynamische Liste, Regeln einzeln hinzufügen) zeigt die Schwellwert-Tabelle **alle 9 Metriken gleichzeitig** als fixe Zeilen. User sieht auf einen Blick alle Einstellungen.

**Datenmodell:** Dasselbe `alert_rules: AlertRule[]` — eine Metrik kann bis zu 2 Regeln haben (absolute + delta = "Beides"). Die Tabelle abstrahiert das intern.

**9 Metriken im Mapping zu bestehenden AlertMetrics:**
- Wind → `wind_gust` / `wind_change`
- Böen → `wind_gust`
- Niederschlag → `precipitation_sum` / `precipitation_change`
- Gewitter → `thunder_level`
- Temperatur → `temperature_min` / `temperature_max` / `temperature_change`
- Nullgrad-Grenze → `snow_line`

Anmerkung: Die Issue-Beschreibung nennt "Regen-Wahrsch.", "Sichtweite", "Schneefall" — diese existieren NICHT im aktuellen `AlertMetric`-Typ. Für #180 wird die Tabelle die **9 bestehenden** Metriken aus `ALERT_METRIC_LABELS` verwenden (keine Modellerweiterung nötig).

**Implementierung:**
- Neue Komponente: `AlertMetricTable.svelte` (Container, alle Zeilen, Speichern-Button)
- Neue Komponente: `AlertMetricRow.svelte` (eine Zeile je Metric)
- Speichern via `PUT /api/trips/{id}` (bestehend)
- **Kein Backend-Change nötig**

### Issue #181: Cooldown + Stille Stunden

**Neue Felder** (noch nicht im Modell):
- `alert_cooldown_minutes: number` — Mindestabstand zwischen Alerts (0 = deaktiviert)
- `alert_quiet_from: string` — Sperr-Zeit ab (z.B. "22:00"), leer = kein Limit
- `alert_quiet_to: string` — Sperr-Zeit bis (z.B. "07:00"), leer = kein Limit

**Änderungen:**
1. **Go** `internal/model/trip.go`: 3 neue Felder in `Trip`-Struct
2. **Go** `internal/handler/trip.go`: `tripUpdateRequest` + Handler Read-Modify-Write
3. **TypeScript** `frontend/src/lib/types.ts`: 3 neue optionale Felder in `Trip`
4. **Python** `src/services/trip_alert.py`: Cooldown + QuietHours aus Trip lesen + anwenden
5. **Frontend**: 2 UI-Cards im Alerts-Tab

### Issue #182: Alert-Preview (Email)

**Konzept:** Zeigt im Alerts-Tab eine visuelle Vorschau, wie eine Alert-E-Mail aussehen würde. Die Vorschau basiert auf den aktuell konfigurierten Regeln.

**Ansatz:** Frontend-Komponente `AlertEmailPreview.svelte` — statische Darstellung der konfigurierten Regeln im E-Mail-Stil (Δ-Bullet-Liste + kompakte Tabelle). Kein neuer Backend-Endpoint nötig (alle Daten bereits im Trip-Objekt).

Alternativ: Backend-rendered via neuen Endpoint (komplex). → Empfehlung: Frontend-only Mock-Preview als MVP.

## Risiken & Überlegungen

1. **Schema-Migration (#181):** Neue Trip-Felder sind optional/nullable → keine Migration nötig, Bestandstrips haben `null` → werden ignoriert (Fallback auf bestehende `throttle_hours`).

2. **UX-Konflikt AlertRulesEditor vs. Tabelle:** Der Edit-Pfad (`/trips/[id]/edit`) nutzt den dynamischen `AlertRulesEditor`, der Alerts-Tab die neue Tabelle. Beide schreiben dieselben `alert_rules`. Konsistenz sicherstellen: Änderungen im Edit-Pfad sollen im Alerts-Tab sichtbar sein (kein eigener State — beides aus `trip.alert_rules`).

3. **Metriken-Scope:** Issue #180 nennt Metriken ("Regen-Wahrsch.", "Sichtweite", "Schneefall"), die nicht im Modell existieren. → Scope auf bestehende 9 Metriken begrenzen, keine Modell-Erweiterung.

4. **QuietHours über Mitternacht (#181):** `from=22:00, to=07:00` bedeutet Sperrung über Mitternacht → die Python-Logik muss das korrekt auswerten.

## Existing Patterns

- **Save-Muster:** `TripEditView.svelte` lädt `trip.alert_rules` als Deep Copy in `$state`, speichert via `PUT /api/trips/{id}` mit `{ alert_rules: ... }`
- **Metric-Labels:** `ALERT_METRIC_LABELS` in `alertMetricLabels.ts` ist Single Source of Truth
- **Design-Tokens:** `docs/reference/design_system.md` + `frontend/src/app.css`
- **DELTA_ONLY_METRICS:** Für Δ-only-Metriken kein `absolute`-Input zeigen
