# Context: gate-scope-1109-1116-1121

**Issues:** #1109 + #1116 + #1121 — drei Bugs in der Scope-Erkennung (`_detect_committed_scope`)
für den Deploy-Gate/Post-Deploy-Selftest. Eigener Gate-Issue-Workflow (kein Feature-Workflow,
siehe `feedback_validator_changes_own_workflow` — Validator ist hier das Deliverable).

## Request Summary

Drei zusammenhängende Lücken in der Scope-Klassifikation (`docs-only` / `frontend-only` /
`backend` / `full-stack`), die entscheidet, ob der Post-Deploy-Selftest läuft. Alle drei
betreffen `.claude/hooks/staging_gate.py` + `.claude/hooks/prod_selftest.py` und teilweise den
Marker `.claude/last_gate_scope.json` (`_e2e_paths.py`).

1. **#1109:** `prod_selftest.py` leitet die Diff-Basis vom **Gate-Marker** ab (letzter
   erfolgreicher `--check`-Lauf), nicht vom **tatsächlich zuletzt live deployten Prod-Commit**.
   Bei mehreren Commits zwischen zwei echten Prod-Deploys (Marker wandert bei jedem
   `--check`-Lauf weiter, auch bei rein staging-seitigen Iterationen) kann der beim
   tatsächlichen Deploy berechnete Scope zu klein ausfallen → Selftest wird für einen echten
   Code-Deploy fälschlich übersprungen.
2. **#1116:** Beschreibt eine Selbstvergiftung des Markers (`marker_sha == HEAD` →
   `HEAD..HEAD`-Diff → fälschlich `docs-only` → wird in den Cache geschrieben →
   `prod_selftest` übernimmt den vergifteten Wert). **Wichtiger Befund (s.u.): dieser exakte
   Mechanismus wurde vermutlich bereits durch #1096 behoben — muss in der Analyse verifiziert
   werden, bevor an #1116 weitergearbeitet wird.**
3. **#1121 (Adversary F003, LOW):** `git diff --name-only <base> HEAD` prüft den Returncode
   nicht. Schlägt der Git-Aufruf fehl (z.B. `<base>` nicht auflösbar), ist `stdout` leer →
   wird identisch zu einem echten leeren Diff behandelt → fälschlich `docs-only`.

## Risks & Considerations

- **Deploy-Gate = kritischster Pfad:** jede Änderung kann Deploys fälschlich blocken (Ops-
  Ärgernis) oder fälschlich durchwinken (Gate-Erosion, die eigentliche Bug-Klasse hier). Im
  Zweifel: blocken/prüfen statt überspringen.
- **Vierte Iteration am selben Mechanismus** — hohes Risiko, dass Fixes sich gegenseitig
  Voraussetzungen entziehen (Marker-Semantik ist bereits doppelt belegt: Diff-Basis für
  nächsten Lauf UND Cache-Key für denselben Commit). #1109 führt ggf. eine DRITTE Bedeutung
  ein („echter Prod-Zustand") — sollte in der Spec sauber von den ersten beiden abgegrenzt
  werden, nicht in denselben Marker gemischt.
- **#1109 ist potenziell Repo-übergreifend** (`henemm-infra/scripts/deploy-gregor-prod.sh`) —
  falls ein neuer Prod-Deploy-Marker dort geschrieben werden muss: Änderung dort separat
  committen, MQ-Nachricht an `infra`-Instanz, nicht in diesem Repo mit-implementieren.
- **`.claude/`-Änderungen klassifizieren selbst als `docs-only`** → das Fix-Deploy dieses
  Workflows wird den eigenen Selftest reflexhaft skippen (identisches Problem wie in
  `fix-1096-gate-scope.md` dokumentiert) — Verifikation muss über hermetische Tests + gezielten
  Staging-Nachweis erfolgen, nicht über den eigenen Post-Deploy-Selftest.
- **Gate-State-Dateien nie per Hand anfassen** (User-Stopp 2026-07-08, `.claude/last_gate_scope.json`
  u.ä.) — Reparatur ausschließlich über Code-Fix + Tests, niemals manuelles Editieren der
  Marker-/Attestation-Dateien.
- **#1121 ist laut Issue-Text LOW/kein Merge-Blocker** — im echten Hauptrepo mit hunderten
  Commits praktisch nicht erreichbar (nur in Ein-Commit-Temp-Repos reproduzierbar). Sauberer,
  isolierter Fix (Returncode-Check + konservativer Fallback), kann unabhängig von #1109/#1116
  in derselben Spec behandelt werden.
- **Eigener Gate-Workflow zwingend:** alle drei Issues betreffen `.claude/hooks/*` — Validator
  ist das Deliverable dieses Workflows selbst. Gold-Standard-Tests müssen sowohl das vergiftete
  Szenario (ablehnen/korrigieren) als auch das korrekte Szenario (akzeptieren) abdecken.

### #1116 — VERDICT: BEREITS BEHOBEN durch #1096 (fab61d76)

Reproduktionsprüfung (hermetisches Temp-Repo, 6 Szenarien inkl. AC-1..AC-4 + F001/F002) zeigt:
der in #1116 beschriebene Poisoning-Pfad ist strukturell nicht mehr möglich. Drei unabhängige
Schutzlagen aus #1096 greifen kombiniert:

1. **F001-Guard** (`_e2e_paths.py:93`, `cached_scope_for_sha()`): ein gecachter `docs-only`-Wert
   wird nie vertraut — kein Cache-Treffer möglich.
2. **F002-Fallback** (`staging_gate.py:126`, `_scope_diff_base()`): Marker zeigt exakt auf HEAD
   → `HEAD~1..HEAD` statt selbstreferenziellem `HEAD..HEAD`-Diff (das war der Kern von #1116).
3. **Write-Guard** (`staging_gate.py:348-350`): ein bereits gecachter Nicht-docs-only-Scope für
   denselben HEAD wird nie mit `docs-only` überschrieben.

`prod_selftest.py` nutzt denselben Shared-Helper (`_e2e_paths.py`) — kein Divergenz-Risiko
zwischen Schreib- und Leseseite mehr.

**Konsequenz:** #1116 wird in diesem Workflow NICHT neu gefixt, sondern mit Verweis auf
`fab61d76` + den Reproduktionstests als „bereits behoben" dokumentiert und geschlossen.

### #1109 — Technischer Ansatz: neuer Deploy-Marker (henemm-infra)

Bestätigt: `last_gate_scope.json` trackt den SHA des letzten *Gate-Laufs*, nicht des zuletzt
*tatsächlich live deployten* Commits — diese können auseinanderlaufen, wenn zwischen zwei
echten Prod-Deploys mehrere Gate-Läufe (z.B. staging-seitige Iterationen) stattfinden.
`e2e_verified.json["verified_commit"]` ist ebenfalls ungeeignet (letzter *staging-verifizierter*
Commit, nicht zwingend der live deployte). `deploy-gregor-prod.sh` schreibt aktuell **keinen**
Marker für „dieser Commit ist jetzt live" (nur Log-Zeile).

**Empfehlung:** neue Marker-Datei `.claude/last_prod_deploy.json` (`{deployed_commit,
deployed_at, status}`), geschrieben von `deploy-gregor-prod.sh` NACH erfolgreichem Smoke-Test
(atomar, exit-code-gesichert). `prod_selftest.py::_scope_diff_base()` liest diese Datei
bevorzugt vor dem Gate-Marker; fehlt sie (Erst-Deploy), Fallback auf bisheriges Verhalten.

**Repo-übergreifend:** Änderung in `henemm-infra/scripts/deploy-gregor-prod.sh` (~5 Zeilen) +
`gregor_zwanzig/.claude/hooks/prod_selftest.py` (~10 Zeilen). Deploy-Script-Änderung separat
committen, MQ-Nachricht an `infra`-Instanz vor Umsetzung.

### #1121 — Technischer Ansatz: Shared-Helper + Returncode-Guard

Betroffene Stellen: `staging_gate.py:161-165` (Haupt-Diff), `staging_gate.py:244-248`
(`_telegram_live_gate`), `prod_selftest.py:434-438` (Haupt-Diff) — je ohne Returncode-Check.

**Fallback-Wert: `"backend"`** (nicht `"full-stack"` — zu pessimistisch; nicht `"docs-only"` —
genau der Bug). Passt zur etablierten Fail-Closed-Philosophie (#1130 F001) und zum Muster aller
anderen git-Aufrufe im Code (`cat-file`, `rev-parse`, `merge-base` prüfen bereits ihren
Returncode).

**Neuer Shared-Helper** `_detect_scope_from_git_diff(base, target, repo_dir)` in `_e2e_paths.py`
— eliminiert ~30 LoC Duplikat-Klassifizierungslogik zwischen beiden Dateien (DRY, konsistent
mit `cached_scope_for_sha`-Familie) und zentralisiert den neuen Returncode-Guard an einer
Stelle statt zwei.

Keine bestehenden Tests für git-Fehler-Szenarien — das ist exakt die Testlücke aus #1121
(Ein-Commit-Temp-Repo mit nicht auflösbarer Diff-Basis).
