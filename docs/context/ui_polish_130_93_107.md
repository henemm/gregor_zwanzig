## Issue #130 — Geänderte Werte Markieren

### Was fehlt
Alert-E-Mails zeigen geänderte Wetterwerte (z. B. "D11 W11 G25") ohne Bezug zur betroffenen Etappe. Außerdem fehlt im Betreff der Etappenname.

### Root Cause
In `src/services/trip_alert.py:_send_alert()` (Zeile 458) wird `format_email()` ohne `stage_name` aufgerufen:
```python
report = self._formatter.format_email(
    segments=weather,
    trip_name=trip.name,
    report_type="alert",   # kein stage_name!
    ...
)
```
→ `trip_report.py:510`: `stage = stage_name or dt.strftime("%d.%m.%Y")` — der Betreff nutzt das Datum statt den Etappennamen.

In der HTML-Mail (`html.py:255-265`) werden Änderungen als `"Segment N (HH:MM–HH:MM) — ..."` gelistet — das ist ein **Zeit-Segment innerhalb eines Tages**, nicht die **Etappe** (z. B. „Tag 3: Vizzavona → Capanelle").

### Betroffene Dateien
| Datei | Relevanz |
|-------|----------|
| `src/services/trip_alert.py:444` | `_send_alert()` — fehlendes `stage_name` |
| `src/formatters/trip_report.py:49` | `format_email()` Signatur + Fallback-Logik |
| `src/output/renderers/email/html.py:255` | `changes_html` — Segment-Label-Render |
| `src/output/renderers/email/helpers.py:445` | `build_segment_label()` — liefert "Segment N (HH:MM-HH:MM)" |
| `src/output/subject.py:140` | `build_email_subject()` — nutzt `token_line.stage_name` |

### Was die Stage-Information liefert
`trip.stages[i]` (aus `app/trip.py`) hat `name` und `date`. `SegmentWeatherData.segment.start_time.date()` kann gegen `stage.date` gematcht werden, um den Etappen-Namen zur Änderung aufzulösen.

### Scope der Änderung
1. `_send_alert()`: Aktive Etappe (Start-Datum == heutiges Datum) ermitteln und `stage_name` an `format_email()` übergeben.
2. `build_segment_label()`: Optional den Etappennamen als Prefix zeigen.

## Issue #107 — Wegpunkt-Felder ohne Label

### Befund
Die ursprünglich gemeldete Datei (`WizardStep2Stages`) existiert heute als `Step2Stages.svelte`. Diese Datei enthält **keine** manuellen Lat/Lon/Höhe-Felder mehr — Step 2 ist jetzt ein GPX-Upload.

Die Felder ohne Labels (placeholder="Lat", "Lon", "Höhe (m)") befinden sich in `frontend/src/lib/components/TripForm.svelte` (Zeilen 107–133). **Dieser Komponent ist toter Code**: Er wird nirgendwo importiert oder geroutet.

### Empfehlung
`TripForm.svelte` löschen (dead code cleanup). Die `LocationForm.svelte` als Referenz zeigt das korrekte Pattern mit `<Label>` für Koordinaten-Felder.

Falls manuelles Waypoint-Hinzufügen im Wizard geplant ist (kein Issue dazu), wäre `LocationForm.svelte` das Vorbild.

## Abhängigkeiten

- **#130 ↔ Trip-Modell**: `trip.stages` muss für Stage-Name-Auflösung verfügbar sein — das ist in `trip_alert.py` bereits der Fall (`trip` ist als Argument vorhanden).
- **#93**: Reine Text-Korrekturen, keine Logik-Abhängigkeiten.
- **#107**: Löschen ohne Risiko (kein Import vorhanden).

## Risiken

- **#130**: Alert-Timing kann komplex sein wenn eine Etappe nicht am heutigen Datum endet — Fallback auf Trip-Name-only bleibt sinnvoll.
- **#93**: Tests mit Playwright/Vitest die auf `data-testid` basieren sind safe — keine funktionalen test-ids ändern sich. aria-label-Änderungen können jedoch E2E-Tests brechen die auf aria-labels selektieren (prüfen).
- **#107**: Zero-Risiko (Dead Code).
