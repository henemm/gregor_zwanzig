ADVERSARY FINDINGS — BUGS AND WARNINGS
================================================================================

FAILED: Live JSON missing channel fields (Backward-Compat NOT auto-applied to disk)
   Problem:
     data/users/default/compare_subscriptions.json (the production file)
     does NOT contain "send_email" or "send_signal" keys for the
     "zillertal-t-glich" subscription. The loader correctly applies defaults
     at read time, but if the file is ever written back via save without
     an explicit edit, the fields ARE now serialized. Until first save,
     the loader must apply defaults on every load — which it does, but the
     on-disk representation is inconsistent with the schema.
   Risk:
     Low for runtime correctness (defaults applied on load).
     Medium for data auditability: a developer inspecting the JSON cannot
     determine the channel config without knowing the code defaults.
   Fix:
     Run a one-time migration or trigger a save on startup to normalize
     existing subscriptions. Alternatively, accept the lazy-migration
     pattern (file updated on next user save) and document it.

WARNING: Both channels unchecked — no UI validation
   Problem:
     Spec explicitly acknowledges: "No validation that at least one channel
     is selected (user can save with both unchecked)."
     With send_email=False and send_signal=False, Run Now produces only a
     warning toast "Kein Kanal konfiguriert oder aktiviert" with no delivery.
     The scheduler silently sends nothing and logs no explicit "no channel"
     warning — it simply does not enter either if-block.
   Risk:
     User confusion: scheduled subscription appears to run but delivers
     nothing, with no error feedback in the UI.
   Recommendation:
     Add a warning log in _execute_subscription() when both flags are False.
     Optionally add UI validation before Save that warns when both are
     unchecked (non-blocking, per spec).

WARNING: Dead imports in subscriptions.py
   Problem:
     Line 25: from outputs.base import OutputConfigError, OutputError
     Neither exception is caught or raised anywhere in the file.
   Risk:
     None at runtime. Misleads readers into thinking error handling exists
     for email/signal output errors in Run Now. The Run Now handler catches
     only bare Exception.
   Recommendation:
     Either remove the imports or add specific exception handling in
     make_run_now_handler().

WARNING: Page heading says "Email Subscriptions" (outdated after F12a)
   Problem:
     subscriptions.py line 51: ui.label("Email Subscriptions")
     After F12a, subscriptions support Signal as well. The heading
     and subtitle ("Automatic ski resort comparison via email") are
     misleading.
   Risk:
     UX confusion only. No functional impact.
   Recommendation:
     Update heading to "Subscriptions" and subtitle to
     "Automatic ski resort comparison via selected channels."

WARNING: SMTP warning banner shown even for Signal-only subscriptions
   Problem:
     subscriptions.py lines 57-67: If SMTP is not configured, a yellow
     warning banner is shown telling the user to configure SMTP first.
     This fires unconditionally regardless of whether any subscription
     uses email. A Signal-only user will always see an SMTP warning.
   Risk:
     Confusing UX for Signal-only users.
   Recommendation:
     Only show the SMTP warning if at least one subscription has
     send_email=True, or suppress it if all subscriptions are Signal-only.

WARNING: Initialization state of CompareSubscription in scheduler
   Problem:
     On first ever run (no compare_subscriptions.json), load_compare_subscriptions()
     returns []. This is safe. However, if a legacy subscription is loaded that
     has never been resaved, send_email defaults to True — meaning an existing
     subscription that the user has not touched will continue to send email,
     which is the correct backward-compatible behavior.
   Risk:
     None. This is the intended behavior per spec. Flagged for completeness.
